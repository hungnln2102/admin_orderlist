# Luồng kiểm tra tài khoản Renew Adobe (form khách hàng)

Tài liệu mô tả luồng **Kiểm tra & Kích hoạt** trên trang Renew Adobe (form nhập email, kiểm tra profile, kích hoạt lại nếu hết hạn).

---

## 1. Tổng quan

- **Vị trí:** Trang Renew Adobe (sidebar) → form "Kiểm tra & Kích hoạt".
- **Input:** Email Adobe (ví dụ: `user@gmail.com`).
- **Hành động:** Bấm **Kiểm tra Profile** → hệ thống gọi API để xác định trạng thái tài khoản và đơn hàng, sau đó hiển thị kết quả. Nếu hết hạn hoặc không tìm thấy, hiện nút **Kích hoạt lại ngay** (gọi API fix-user).

---

## 2. Luồng kiểm tra (khi bấm "Kiểm tra Profile")

### Bước 1: Gọi API song song

Frontend gọi **hai API cùng lúc**:

| API | Method | Mục đích |
|-----|--------|----------|
| `GET /api/renew-adobe/accounts/lookup?email={email}` | GET | Tìm tài khoản admin Adobe chứa email này (theo `account.email` hoặc `users_snapshot`). Trả về `account` (org_name, license_status, ...) hoặc 404. |
| `GET /api/renew-adobe/user-orders` | GET | Lấy toàn bộ đơn hàng (order_code, information_order = email, expiry_date, ...) để so sánh **thời hạn sử dụng**. |

Cả hai đều dùng `credentials: "include"` (cookie/session nếu có).

### Bước 2: Xử lý kết quả lookup

- **404 hoặc không có `data.account`:**
  - Hiển thị: **Profile hết hạn** + *"Không tìm thấy tài khoản Adobe cho email này. Bấm nút bên dưới để kích hoạt."*
  - Hiện nút: **Kích hoạt lại ngay**
  - Kết thúc luồng.

- **Lookup lỗi (res không ok):**
  - Hiển thị thông báo lỗi từ server hoặc mặc định.
  - Không hiện nút kích hoạt (tùy implementation).

- **Có account:** Lấy `org_name`, `license_status` (chuẩn hóa lowercase).

### Bước 3: Kiểm tra license_status (tài khoản admin)

- Nếu `license_status` là **expired** hoặc **unknown**:
  - Hiển thị: **Profile hết hạn** + *"Gói Adobe đã hết hạn. Bấm nút bên dưới để được cấp lại profile mới."*
  - Hiện nút: **Kích hoạt lại ngay**
  - Kết thúc.

- Nếu còn gói (ví dụ: paid, hoặc bất kỳ giá trị khác không phải expired/unknown): Tiếp bước 4.

### Bước 4: Kiểm tra thời hạn sử dụng (đơn hàng)

- Nếu gọi `user-orders` thành công:
  - Lọc danh sách đơn theo `information_order` trùng với email (không phân biệt hoa thường).
  - Lấy đơn có **expiry_date** mới nhất (đại diện cho gói đang dùng).
  - So sánh `expiry_date` với **ngày hiện tại** (theo múi giờ client, chỉ so ngày):
    - Nếu **expiry_date < hôm nay** → coi là **đã hết hạn sử dụng**.
- Nếu **đã hết hạn sử dụng**:
  - Hiển thị: **Profile hết hạn** + *"Đã hết hạn sử dụng (Hạn: DD/MM/YYYY). Bấm nút bên dưới để kích hoạt lại."*
  - Hiện nút: **Kích hoạt lại ngay**
  - Kết thúc.

### Bước 5: Trạng thái bình thường

- Điều kiện: Đã có account, license còn gói, và (không có đơn nào cho email **hoặc** có đơn nhưng chưa hết hạn).
- Hiển thị: **Profile đang hoạt động bình thường** + hiển thị **org_name** (tên Profile/Org).
- Không hiện nút Kích hoạt.

## 3. Tóm tắt kết quả hiển thị

## 4. Luồng Kích hoạt (khi bấm "Kích hoạt lại ngay")

- **API:** `POST /api/renew-adobe/fix-user`
- **Body:** `{ "email": "<email đã nhập>" }`
- **Credentials:** `include`

**Backend (tóm tắt):** Tìm tài khoản admin còn gói và còn slot (ưu tiên ít slot trống nhất), add user vào tài khoản đó và gắn product (Adobe API). Trả về success + message + profile (org_name).

**Frontend:**
- Thành công: Hiển thị **Chuyển profile thành công** + org_name + nút "Hướng dẫn đăng nhập lại Team".
- Thất bại: Hiển thị lỗi từ server (ví dụ: "Không có tài khoản nào còn slot.").

## 6. API liên quan (backend admin_orderlist)

| Endpoint | File / Ghi chú |
|----------|----------------|
| `GET /api/renew-adobe/accounts/lookup?email=` | RenewAdobeController: tìm account theo email (email admin hoặc trong users_snapshot). |

## 7. Tại sao phải duyệt qua tài khoản admin để lấy tài khoản cần check?

**Câu hỏi:** Tại sao không lấy trực tiếp “user” cần check, mà lại phải tìm **tài khoản admin** chứa user đó?

**Trả lời ngắn gọn:** Trong hệ thống hiện tại, **không có bảng/khóa riêng** lưu quan hệ “email user → tài khoản admin”. Thông tin “user này thuộc tài khoản nào” **chỉ nằm bên trong** từng tài khoản admin, trong cột `users_snapshot` (JSON). Vì vậy muốn biết “tài khoản cần check” cho một email thì phải **duyệt qua các tài khoản admin** và mở `users_snapshot` ra xem.

**Chi tiết:**

1. **Mô hình dữ liệu hiện tại**
   - Có bảng **tài khoản admin** (ví dụ `accounts_admin`): mỗi dòng = 1 tài khoản Adobe (email admin, org_name, license_status, …).
   - Mỗi tài khoản có cột **`users_snapshot`**: một JSON array chứa danh sách user (email, name, product, …) **đang nằm trong tài khoản đó** — dữ liệu này được cập nhật khi “check” (scrape từ Adobe).
   - **Không** có bảng kiểu “user → account_id” hay index riêng: tức là không có chỗ nào lưu trực tiếp “email `user@gmail.com` thuộc account_id 5”.

2. **Vì vậy lookup phải làm gì?**
   - Input: **email user** (ví dụ email khách nhập ở form).
   - Câu hỏi cần trả lời: “User này đang nằm trong **tài khoản admin nào**?” (để lấy org_name, license_status, …).
   - Cách duy nhất với dữ liệu hiện tại:  
     - Bước 1: thử tìm theo **email = email admin** (trường hợp user chính là admin).  
     - Bước 2: nếu không trùng, **duyệt lần lượt từng tài khoản admin**, parse `users_snapshot`, kiểm tra xem có phần tử nào có `email` trùng với email cần tìm không → tìm được “tài khoản admin chứa user này”.

3. **Tại sao cần “tài khoản admin” chứ không chỉ “user”?**
   - Form cần hiển thị **org_name** (tên Profile/Org) và **license_status** (còn gói / hết hạn) — đây là thông tin **của tài khoản admin**, không lưu tách ở “user”.
   - Trạng thái “còn gói / hết hạn” được xác định theo gói của **tài khoản admin** đó (check product ở trang Adobe, cập nhật license_status). Vì vậy phải có **tài khoản admin** thì mới có đủ dữ liệu để “check” đúng nghĩa.

**Tóm lại:** Phải duyệt qua tài khoản admin vì quan hệ “email user → tài khoản” **chỉ tồn tại bên trong** `users_snapshot` của từng tài khoản, chưa có bảng/index tra ngược. Nếu sau này bổ sung bảng hoặc cache dạng “user_email → account_id” (đồng bộ khi lưu snapshot), có thể đổi lookup thành một truy vấn trực tiếp theo email user mà không cần duyệt toàn bộ tài khoản admin.

## 8. Giữa key_active và accounts_admin cần bảng gì ở giữa để đối chiếu nhanh?

**Câu hỏi:** Giữa bảng `key_active` (order_auto_keys) và `accounts_admin` cần bảng gì ở giữa để có thể đối chiếu nhanh (lookup email user → tài khoản admin mà không duyệt hết snapshot)?

**Gợi ý: bảng ánh xạ “user email → account”**

Thêm một bảng **trung gian** lưu quan hệ **email user → account_id**, cập nhật mỗi khi lưu/check `users_snapshot` của một tài khoản admin. Khi đó lookup chỉ cần 1 truy vấn theo email, không cần duyệt toàn bộ `accounts_admin`.

**Ví dụ cấu trúc:**

| Bảng (ví dụ tên) | Mục đích |
|------------------|----------|
| `renew_adobe_user_account` (hoặc tương đương) | Ánh xạ nhanh: user_email → account_id (và có thể thêm order_code nếu cần đối chiếu với đơn). |

**Cột gợi ý:**

- `user_email` (VARCHAR, unique hoặc unique cùng account_id tùy logic) — email user (lowercase, trim).
- `account_id` (FK → accounts_admin.id) — tài khoản admin chứa user này.
- `order_code` (VARCHAR, nullable, optional) — mã đơn từ order_auto_keys/order_list nếu muốn gắn “user này đang dùng đơn nào” (có thể lấy từ order_list qua information_order = email).
- `updated_at` (timestamp) — lần cập nhật cuối (khi sync từ snapshot).

**Luồng đồng bộ:**

1. Mỗi khi job check tài khoản cập nhật `users_snapshot` của một `accounts_admin`:
   - Xóa (hoặc đánh dấu xóa) các dòng cũ có `account_id = id` đó.
   - Với mỗi user trong snapshot (có email), insert/upsert vào bảng trung gian: `(user_email, account_id, updated_at)` (và order_code nếu có).
2. Khi một user bị xóa khỏi tài khoản (job xóa user), xóa hoặc cập nhật dòng tương ứng trong bảng trung gian.

**Quan hệ với key_active / order_list:**

- `key_active.order_auto_keys` ↔ `order_list`: đã có (order_code = id_order) → có email qua `order_list.information_order`, expiry qua `order_list.expired_at`.
- Bảng trung gian **không thay thế** key_active hay order_list; nó chỉ bổ sung **tra cứu nhanh**: email user → account_id. Đối chiếu “đơn hàng – tài khoản” vẫn có thể qua order_list (email) + bảng trung gian (email → account_id).

**Kết quả:** Lookup API có thể: `SELECT account_id FROM renew_adobe_user_account WHERE user_email = ?` rồi `SELECT * FROM accounts_admin WHERE id = ?`, thay vì duyệt toàn bộ tài khoản admin và parse `users_snapshot`.

## 9. Có nên bắt buộc có Key (đơn hàng) để user sử dụng được hệ thống?

**Câu hỏi:** Có nên cần thiết có Key (mã đơn / đơn hàng trong hệ thống) để user có thể sử dụng được hệ thống (check + kích hoạt)?

**Hiện trạng:**

- **Check (lookup + user-orders):** User chỉ cần nhập email. Không bắt buộc nhập mã đơn hay chứng minh đã mua hàng.
- **Kích hoạt (fix-user):** Backend **không** kiểm tra email có nằm trong `order_list` / `key_active` hay không. Bất kỳ email nào gửi lên đều có thể được add vào tài khoản admin còn slot.  
→ Về mặt kỹ thuật, **hiện tại không cần Key** để “dùng” được (check + kích hoạt).

**Nên hay không (gợi ý):**

- **Nếu mục tiêu là chỉ phục vụ khách đã mua đơn:** **Nên** yêu cầu có Key (đơn hàng hợp lệ). Cụ thể:
  - Chỉ cho phép **kích hoạt** khi email có đơn trong hệ thống (order_list / key_active) và đơn **chưa hết hạn** (expiry_date ≥ hôm nay).  
  - Có thể vẫn cho **check** (xem trạng thái) mà không cần Key; chỉ khi bấm “Kích hoạt lại” mới bắt buộc kiểm tra đơn.
- **Cách làm:** Trong API `POST /api/renew-adobe/fix-user`, trước khi add user vào tài khoản admin, thêm bước:
  - Tra order_list (hoặc user-orders) theo email (`information_order` = email).
  - Nếu không có đơn **hoặc** mọi đơn đều đã hết hạn → trả 403/400, message kiểu: *"Chỉ khách hàng có đơn hàng còn hiệu lực mới được kích hoạt. Vui lòng mua đơn hoặc gia hạn."*
  - Nếu có ít nhất một đơn còn hiệu lực → tiếp tục logic add user như hiện tại.

**Tóm lại:**

| Cách làm | Ưu điểm | Nhược điểm |
|----------|--------|-------------|
| **Không bắt Key** (như hiện tại) | Đơn giản, bất kỳ email nào cũng có thể thử kích hoạt. | Dễ lạm dụng: người chưa mua vẫn có thể được cấp slot. |
| **Bắt Key (đơn còn hiệu lực)** | Chỉ khách có đơn mới được kích hoạt → đúng đối tượng, tránh lạm dụng. | Cần sửa backend fix-user để kiểm tra order_list/expiry trước khi add user. |

**Kết luận:** Về mặt nghiệp vụ thường **nên** coi Key (đơn hàng còn hiệu lực) là điều kiện cần để được **kích hoạt**; có thể giữ **check** mở cho mọi người (chỉ xem trạng thái), còn **kích hoạt** thì chỉ cho email có đơn hợp lệ.

## 10. Dùng mã đơn hàng làm Key active + chỉ một số sản phẩm dùng Renew Adobe — Thiết kế database

**Yêu cầu:**
- Dùng **mã đơn hàng** (order code, tức `order_list.id_order`) làm Key active luôn, không tạo key riêng.
- Chỉ **một số sản phẩm** (variant) dùng hệ thống Renew Adobe, không phải toàn bộ.
- Cần thiết kế lại DB cho phù hợp: có còn cần bảng `key_active` không, và làm sao biết đơn nào thuộc Renew Adobe.

### 10.1. Có còn cần bảng `key_active` (order_auto_keys) không?

- **Cho riêng luồng Renew Adobe:** **Không bắt buộc.** Nếu “key active” = chính mã đơn hàng thì ta chỉ cần `order_list` + một cách để biết “đơn này có dùng Renew Adobe không” (ví dụ: sản phẩm thuộc danh sách Renew Adobe). Khi đó:
  - “Đơn active cho Renew Adobe” = đơn có trong `order_list`, **và** `id_product` (variant_id) nằm trong danh sách sản phẩm dùng Renew Adobe, và (nếu cần) chưa hết hạn, status hợp lệ.
  - Không cần lưu thêm bản ghi tương ứng trong `key_active.order_auto_keys` cho Renew Adobe.
- **Nếu `key_active` đang phục vụ hệ thống khác** (ví dụ sản phẩm khác, system_code khác): giữ bảng `key_active` cho các hệ thống đó; chỉ **Renew Adobe** là không dùng `key_active`, mà dùng `order_list` + bảng whitelist sản phẩm (xem dưới).

### 10.2. Làm sao biết “chỉ một số sản phẩm dùng Renew Adobe”?

Cần một **danh sách “sản phẩm Renew Adobe”** (theo variant hoặc product). Hai hướng thường dùng:

**Cách 1: Bảng whitelist (nên dùng)**

- Bảng **`system_automation.product_system`** (variant_id, system_code): ánh xạ sản phẩm thuộc hệ thống nào; job hàng loạt lấy variant_id theo system_code rồi query order_list.
- Mục đích: đánh dấu **variant nào** (hoặc product nào) được dùng cho Renew Adobe.

Ví dụ cấu trúc (theo **variant** — vì đơn lưu `id_product` = variant_id):

| Bảng `renew_adobe_product` | |
|----------------------------|--|
| `variant_id` (PK, FK → product.variant.id) | Variant thuộc sản phẩm Renew Adobe. |
| `created_at` (optional) | Ngày thêm vào danh sách. |

- Một đơn thuộc Renew Adobe khi và chỉ khi: `order_list.id_product` ∈ (SELECT variant_id FROM renew_adobe_product).
- Khi thêm/bỏ sản phẩm khỏi Renew Adobe: chỉ cần insert/delete trong bảng này, không đụng vào `order_list` hay `key_active`.

**Cách 2: Cột trên bảng variant/product**

- Thêm cột kiểu `use_renew_adobe BOOLEAN DEFAULT false` vào bảng variant (hoặc product).
- Đơn thuộc Renew Adobe khi: `order_list.id_product` trỏ tới variant có `use_renew_adobe = true`.
- Ưu: không thêm bảng. Nhược: gắn chặt logic Renew Adobe vào schema sản phẩm chung.

Gợi ý: dùng **Cách 1** (bảng whitelist) để tách rõ “hệ thống Renew Adobe” và dễ bảo trì.

### 10.3. Thiết kế database tổng thể (tóm tắt)

- **Giữ nguyên:**  
  - `order_list`: id, id_order (mã đơn hàng), id_product (variant_id), information_order (email), customer, expired_at, status, …  
  - `accounts_admin` (Renew Adobe): như hiện tại.  
  - `key_active.order_auto_keys`: **giữ nếu** vẫn dùng cho hệ thống/key khác (không dùng cho Renew Adobe).

- **Thêm mới:**  
  - **`renew_adobe_product`**: `variant_id` (PK, FK variant), (optional) `created_at`.  
  - (Đã bàn ở mục 8) **`renew_adobe_user_account`** (optional): `user_email`, `account_id`, (optional) `order_code`, `updated_at` — dùng cho tra cứu nhanh email → account, không thay thế key.

- **Quan hệ:**  
  - “Key active” cho Renew Adobe = **mã đơn hàng** = `order_list.id_order`.  
  - Đơn “được dùng” trong Renew Adobe khi: `order_list.id_product` ∈ `renew_adobe_product.variant_id` (và có thể thêm điều kiện status, expired_at).

- **Không bắt buộc với Renew Adobe:**  
  - Không cần thêm bản ghi tương ứng trong `key_active.order_auto_keys` cho từng đơn Renew Adobe. Có thể bỏ qua bảng này hoàn toàn cho luồng Renew Adobe.

### 10.4. Luồng xử lý (sau khi thiết kế)

- **Lấy danh sách đơn Renew Adobe (thay cho user-orders cũ dựa trên key_active):**  
  `SELECT order_list.* FROM order_list`  
  `INNER JOIN renew_adobe_product ON order_list.id_product = renew_adobe_product.variant_id`  
  (và lọc status, expired_at nếu cần). Có thể vẫn trả về dạng: order_code, information_order, customer, expiry_date, status.

- **Kiểm tra “user có Key (đơn) hợp lệ” khi kích hoạt:**  
  - Input: email (và có thể mã đơn hàng nếu muốn bắt nhập).  
  - Tìm đơn: `order_list` WHERE `information_order` = email AND `id_product` IN (SELECT variant_id FROM renew_adobe_product) AND `expired_at` >= today (và status hợp lệ).  
  - Nếu có ít nhất một đơn thỏa mãn → cho phép gọi fix-user (add user vào account). Không có đơn hợp lệ → 403/400 “Chỉ khách có đơn Renew Adobe còn hiệu lực mới được kích hoạt”.

- **Check trạng thái:** Có thể vẫn dùng lookup theo email + so expiry từ tập đơn lấy từ `order_list` + `renew_adobe_product` (không cần key_active).

### 10.5. Tóm tắt

| Thành phần | Có cần? | Ghi chú |
|------------|--------|--------|
| **Mã đơn hàng làm Key** | Có | Key active = `order_list.id_order`, không tạo key riêng. |
| **Bảng `key_active.order_auto_keys`** | Không (cho Renew Adobe) | Có thể bỏ dùng cho Renew Adobe; giữ nếu hệ thống khác vẫn cần. |
| **Bảng `renew_adobe_product`** | Có (nên) | Whitelist variant_id dùng Renew Adobe → chỉ một số sản phẩm. |
| **Bảng `renew_adobe_user_account`** | Tùy chọn | Tra cứu nhanh email → account_id. |
| **order_list + renew_adobe_product** | Đủ | Xác định đơn nào thuộc Renew Adobe và còn hiệu lực. |

## 11. Troubleshooting — Khi toàn bộ tính năng Renew Adobe bị lỗi

Khi **check tài khoản admin**, **thêm user**, **xóa user** và **thêm product (product-system)** đều báo lỗi, kiểm tra theo thứ tự sau.

### 11.1. Lỗi 401 — Chưa đăng nhập (nguyên nhân thường gặp nhất)

- **Tất cả API Renew Adobe** (`/api/renew-adobe/*`) đều đi qua `authGuard`: cần **session đăng nhập admin**.
- Nếu **chưa đăng nhập** hoặc **session hết hạn**, mọi request trả về **401** với body `{ "error": "Không có quyền truy cập" }`.

**Cách kiểm tra:**

1. Mở DevTools (F12) → tab **Network**.
2. Thực hiện một thao tác (ví dụ: Check tài khoản, Thêm user, Xóa user, hoặc Thêm product trong Product System).
3. Chọn request tương ứng (ví dụ `check`, `add-user`, `auto-delete-users`, `product-system`).
4. Xem **Status**: nếu là **401** → lỗi do **chưa đăng nhập** hoặc **hết phiên**.

**Cách xử lý:** Đăng nhập lại vào trang admin (auth), sau đó thử lại Renew Adobe.

### 11.2. Lỗi 4xx/5xx — Backend / Adobe API

- Nếu **không phải 401** (ví dụ 400, 404, 500), lỗi đến từ **backend** hoặc **Adobe** (login/API).
- Response body thường có `error` hoặc `message` (ví dụ: "Check thất bại.", "Login thất bại", "Không có access token").

**Cách kiểm tra:**

1. Trong Network, xem **Response** body của request lỗi.
2. Xem log backend (console hoặc file log): controller ghi `[renew-adobe] ...`; service adobe-http ghi `[adobe-http] ...`.
3. Các nguyên nhân thường gặp:
   - **Thiếu/ sai email hoặc password_enc** của tài khoản admin trong DB.
   - **Adobe thay đổi API** hoặc chặn request (cần cập nhật adobe-http hoặc dùng Playwright).
   - **Hết slot** (add user): "Không có tài khoản nào còn slot." hoặc "Không đủ slot".
   - **DB**: thiếu bảng/ cột (ví dụ `system_automation.accounts_admin`, `alert_config`, `mail_backup_id`) → chạy đủ migration.

### 11.3. Tóm tắt luồng và nơi dễ lỗi

| Tính năng | Endpoint | Đi qua authGuard? | Phụ thuộc Adobe API? |
|-----------|----------|-------------------|------------------------|
| Check tài khoản admin | `POST /api/renew-adobe/accounts/:id/check` | Có | Có (login + getOrg + getProducts + getUsers) |
| Thêm user | `POST /api/renew-adobe/accounts/:id/add-user` | Có | Có (login + addUsers) |
| Xóa user | `POST /api/renew-adobe/accounts/:id/auto-delete-users` | Có | Có (login + removeUser) |
| Thêm product (variant vào Renew Adobe) | `POST /api/renew-adobe/product-system` | Có | Không (chỉ insert DB) |

- **Nếu cả bốn đều lỗi giống nhau (401):** gần như chắc chắn là **chưa đăng nhập**.
- **Nếu chỉ check / add / delete user lỗi** còn **thêm product-system vẫn OK:** lỗi ở **Adobe login hoặc API** (credentials, API Adobe, hoặc adobe-http).
- **Nếu thêm product-system cũng lỗi** cùng kiểu với các API kia → thường là **401** (auth).

### 11.4. Đổi IP khi bị Adobe block (sau 5 lần đăng nhập sai)

Adobe có thể **block IP** sau vài lần đăng nhập thất bại. Để dùng **IP khác** (proxy), cấu hình biến môi trường:

| Biến môi trường | Mô tả |
|-----------------|--------|
| **ADOBE_PROXY** | Proxy dùng riêng cho luồng Adobe (ưu tiên cao nhất). |
| **ADOBE_HTTP_PROXY** | Cùng ý nghĩa, dự phòng. |
| **HTTPS_PROXY** | Proxy HTTPS (nhiều tool dùng chung). |
| **HTTP_PROXY** | Proxy HTTP. |

**Định dạng:** URL proxy, ví dụ:

- `http://proxy.example.com:8080`
- `http://user:password@proxy.example.com:8080`
- `socks5://host:1080` (nếu Playwright/axios hỗ trợ)

**Cách dùng:**

1. Cấu hình trong `.env` (backend) hoặc môi trường khi chạy:
   ```env
   ADOBE_PROXY=http://your-proxy-host:port
   ```
   Nếu proxy cần đăng nhập:
   ```env
   ADOBE_PROXY=http://username:password@your-proxy-host:port
   ```

2. Restart backend (hoặc process chạy adobe-http) để áp dụng.

3. Mọi request **HTTP (axios)** và **Playwright (browser)** trong luồng Adobe (check, add user, delete user, auto-assign, …) sẽ đi qua proxy → Adobe thấy IP của proxy, không còn IP server bị block.

**Tắt proxy:** Đặt `ADOBE_PROXY=` (rỗng) hoặc `ADOBE_PROXY=0` rồi restart.

#### Test bằng proxy chạy trên máy local (IP của bạn)

Để Adobe thấy **đúng IP máy bạn** khi test:

1. **Chạy proxy trên máy bạn** (cùng máy với backend hoặc máy riêng):
   ```bash
   cd admin_orderlist/backend
   npm run proxy:local
   ```
   Hoặc: `node scripts/local-proxy.js`  
   Proxy listen mặc định: `http://0.0.0.0:3128` (port có thể đổi bằng `PROXY_PORT=8888`).

2. **Cấu hình backend dùng proxy:**
   - Backend chạy **cùng máy** với proxy → trong `.env`:
     ```env
     ADOBE_PROXY=http://localhost:3128
     ```
   - Backend chạy **máy khác** (VD server), proxy chạy trên máy bạn → mở port 3128 (firewall), trong `.env` trên server:
     ```env
     ADOBE_PROXY=http://<IP_MAY_BAN>:3128
     ```
   Để biết IP máy bạn: Windows `ipconfig`, macOS/Linux `ip a` hoặc `ifconfig` (xem IPv4 của card mạng nội bộ).

3. **Restart backend**, sau đó thử Check tài khoản / Add user. Request sẽ đi: Backend → proxy (máy bạn) → Adobe → Adobe thấy IP máy bạn.
